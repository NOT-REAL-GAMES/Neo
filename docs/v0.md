# Neo v0 Contract

Neo v0 is a proof that a small, explicit graphics-kernel language can move from
source text to a real NVIDIA GPU image without OpenGL-style hidden state.

## Required Workflow

- `neo doctor` reports CUDA driver and NVRTC availability.
- `neo compile <file.neo> --out <file.cu>` lowers Neo source to CUDA C.
- `neo run <file.neo> --out <file.png>` compiles with NVRTC, launches the `image`
  kernel, downloads RGBA pixels, and writes a PNG.
- `neo-live-window <file.neo>` opens a live no-interop Win32 presenter for the
  animated image ABI. Use `--frames N` or `--seconds N` for bounded stress runs.
  Use `--mode kernel-throughput` to count completed kernels separately from
  sampled readbacks and visible presents.

## Language Surface

- Source files contain one or more `kernel fn` declarations.
- Kernel parameters are explicit and typed.
- Pointer buffers use explicit address spaces such as `global u8* pixels`.
- Kernel bodies use the v0 C-like subset plus Neo conveniences:
  - `let name: type = expr;`
  - `thread_id()`, `block_id()`, `block_dim()`, `grid_dim()`
  - `as_u8`, `as_i32`, `as_u32`, `as_f32`
  - `vec2f`, `vec3f`, `vec4f`

## Runtime Surface

- `Context::new_default_device()`
- `Module::from_neo_source(source, entrypoints)`
- `DeviceBuffer<T>::new`, `upload`, `download`, `download_into`,
  `download_into_pinned`
- `PinnedHostBuffer<T>::new` via `Context::alloc_pinned`
- `Kernel::launcher()` and explicit launch dimensions
- `ImageBuffer::save_png(path)`

## Non-Goals

- Vulkan, DirectX, or OpenGL interop.
- A full expression type checker.
- Scene/material/render-graph authoring.
- Cross-vendor GPU support.

## Live Window Extension

The live example is the first step beyond batch PNG output. It launches Neo into
a CUDA-owned device buffer, writes presenter-native BGRA8 pixels, downloads into
a reused host buffer, and presents with `SetDIBitsToDevice`. The app logs FPS
and per-frame stage timings once per second and keeps the last good compiled
module active when hot reload fails.

The kernel-throughput mode keeps the visible no-interop window but samples
readback/presentation. Its `kernel_fps` value counts completed CUDA image
kernels, not sampled readbacks or visible presents.
